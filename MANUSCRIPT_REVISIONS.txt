General Objective

To design, develop, and evaluate TASKCHAMP, a mobile-based student planner and
task tracker that applies a deterministic urgency-importance prioritization
algorithm combined with an AI-assisted recommendation and grade-forecasting
model to reduce missed deadlines, improve scheduling effectiveness, and
centralize academic and personal task management for college students at the
University of Cebu Lapu-Lapu and Mandaue.

Specific Objectives

1. Address the identified challenges in academic task management by:
   1.1 Tracking and reducing the incidence of missed deadlines through a
       dedicated metric that aggregates currently-overdue tasks and
       historically late completions, displayed as an on-time-rate percentage
       on the analytics dashboard.
   1.2 Implementing effective scheduling and prioritization through a Smart
       Priority algorithm that combines deadline-driven urgency (0–50) and
       stakes-driven importance (0–50), explicitly excluding subjective
       difficulty ratings to prevent false-urgent labeling.

2. Provide a unified workspace for academic and personal task management by
   enabling users to:
   2.1 Create, view, update, and delete both academic and personal tasks,
       distinguished by an explicit Academic / Personal toggle that
       conditionally exposes course-related fields.
   2.2 Visualize tasks across three views — list, Kanban board (drag-and-drop
       status), and a centralized calendar that merges tasks with academic
       events.
   2.3 Record focused study time using a built-in Pomodoro-style timer with
       customizable session durations (15, 25, 45, 60, or any value 1–180
       minutes) and break / long-break cycles.

3. Implement an Academic Hub module that allows students to:
   3.1 Manage personal subjects (course code, professor, credits, current
       grade, target grade) with per-task linking and grade-weight tracking.
   3.2 Receive AI-generated task recommendations and grade forecasts through
       Task AI, presenting the top three priority items and a per-course
       predicted final grade.

4. Implement an Administrator module that allows authorized staff to:
   4.1 Maintain the UCLM degree-program registry that powers the registration
       dropdown.
   4.2 View, edit, promote, or remove student records.
   4.3 Monitor system-wide statistics (total students, active programs, recent
       signups).

5. Implement secure user account and profile management, including:
   5.1 Email-and-password registration capturing the student's degree program
       (BSCS, BSIT, etc.) and education level (1st year, 2nd year, …,
       Master's, PhD).
   5.2 Authentication backed by Supabase Auth with row-level security ensuring
       each user can only access their own data.
   5.3 Profile editing for name, course, and education level.

6. Integrate context-aware notifications (push and in-app) to remind users of
   upcoming, due-soon, and overdue tasks, and to confirm completion of focused
   study sessions.

7. Design a user-friendly mobile interface that follows Material Design
   principles, supports both academic and personal workflows in a single
   application, and is suitable for the Android devices most commonly used by
   UCLM students.

--- END PASTE ---

--- BEGIN PASTE ---

Algorithms Used in the System

The system implements two algorithms that directly fulfill specific objectives
1.2 (effective scheduling and prioritization) and 3.2 (AI-generated grade
forecasting).

A. Smart Priority Algorithm (Task Prioritization)

Purpose. Rank a student's pending tasks so they can focus on what matters next,
blending two orthogonal signals: how soon something is due (urgency) and how
much it is worth (importance). The algorithm is deterministic and explainable,
returning a 0-to-100 score per task.

Formula:

    SmartPriority(t) = UrgencyScore(t) + ImportanceScore(t),  range [0, 100]

Urgency Score (0–50) — derived strictly from time-to-deadline:

    Overdue                     → 50
    Due ≤ 1 day                 → 45
    Due ≤ 3 days                → 35
    Due ≤ 7 days                → 25
    Due ≤ 14 days               → 15
    > 14 days                   → 5
    No due date                 → 0

Importance Score (0–50) — for academic tasks:

    basePriority = { high: 20, medium: 12, low: 6 }[task.priority]
    typeBonus    = { exam: 15, project: 12, assignment: 10,
                     study: 8, reading: 6, other: 5 }[task.taskType]
    weightBonus  = min(task.gradeWeight / 5, 10)

    Importance   = clamp(basePriority + typeBonus + weightBonus, 0, 50)

For personal tasks, only basePriority applies.

Note on difficulty. Difficulty is captured for effort estimation in the UI
but is intentionally excluded from the priority score. An earlier version
multiplied importance by 1.2 for hard tasks, which caused low-stakes-but-
difficult tasks to outrank deadline-critical assignments — falsely tagging
them "urgent." This was corrected based on panel feedback.

Worked Example. A high-priority CS101 final exam worth 40% of the grade,
due in 5 days:

    Urgency       = 25                       (≤ 7 days)
    basePriority  = 20                       (high)
    typeBonus     = 15                       (exam)
    weightBonus   = min(40 / 5, 10) = 8
    Importance    = 20 + 15 + 8 = 43
    SmartPriority = 25 + 43 = 68 / 100

This task would appear near the top of the student's priority list.


B. Grade Forecast Algorithm

Purpose. Estimate the student's final percentage grade per course before the
semester ends, supporting self-monitoring and early intervention.

Two-tier approach.

Tier 1 — Deterministic baseline:

    avgEarnedGrade = Σ(weight × grade / 100) ÷ Σ(weight)
                       (over completed-and-graded tasks)

    pendingWeight  = Σ weight                     (over incomplete tasks)
    earnedWeight   = Σ weight                     (over completed-and-graded tasks)
    unknownWeight  = max(100 − earnedWeight − pendingWeight, 0)

    predictedFinal = (avgEarnedGrade × earnedWeight) +
                     (avgEarnedGrade × pendingWeight) +
                     (avgEarnedGrade × unknownWeight)

Plain English: "What you've earned, plus your historical average applied to
what's still left."

Tier 2 — AI-augmented refinement:
The deterministic baseline is sent to Task AI with deadline pressure and recent
trend data. The AI may adjust the estimate by ±3–5 points if it detects a
strong improving / declining pattern, historical underperformance on a specific
assessment type, or workload pressure that risks rushed work. The output is
presented as a range (e.g., "85–89%") with the deterministic value as the
midpoint.

Worked Example. CS101 midway through the semester:

    Task         Weight    Grade
    Quiz 1       10%       88
    Quiz 2       10%       92
    Midterm      30%       85
    Final        40%       (pending)
    Project      10%       (pending)

    Earned weight contribution = 10×0.88 + 10×0.92 + 30×0.85 = 43.5
    Average earned grade        = 43.5 / 50 = 87%
    Pending weight              = 50
    predictedFinal              = 43.5 + (0.87 × 50) = 87

    Predicted final grade ≈ 87%

--- BEGIN PASTE ---

Evaluation of Task Prioritization

The Smart Priority algorithm is evaluated on five metrics across a two-week
field trial with 10–15 UCLM students who use TASKCHAMP for all academic tasks
during the trial.

  Metric                          Definition                                Target
  --------------------------      --------------------------------------    ----------
  Precision@3                     Of the three tasks the algorithm ranks    ≥ 0.70
                                  highest, how many appear among the
                                  user's first three actual completions?
  Mean Time-to-Completion (top-3) Average days from creation to             Lower than for
                                  completion for tasks ranked top three.    un-prioritized tasks
  Missed Deadline Rate            (overdue + late completions) /            ≤ 0.20
                                  total tasks with deadlines
  On-Time Rate                    Tasks completed on or before deadline /   ≥ 0.80
                                  total deadline-bearing tasks
  User Agreement (Likert)         "The recommended priority order matches   Mean ≥ 4.0 / 5
                                  what I would have chosen myself."

Data is captured automatically via the tasks table fields (smart_priority,
due_date, completed, updated_at) and supplemented by a post-trial Likert
survey administered to all participants.


Evaluation of Grade Forecasting

Forecasts are captured at week 1, week 4, and week 8 of the semester, then
compared to the actual final grade reported by the student.

  Metric                          Definition                                Target
  --------------------------      --------------------------------------    ----------
  MAE (Mean Absolute Error)       Σ |predicted − actual| / N                ≤ 5
                                                                            percentage points
  RMSE (Root Mean Squared Error)  √(Σ(predicted − actual)² / N)             ≤ 7
  Direction Accuracy              (# correctly predicted pass/fail) / N     ≥ 0.85
  Forecast Stability              Standard deviation of predictions for     ≤ 5
                                  the same course across the three
                                  checkpoints

Reporting MAE and RMSE per checkpoint demonstrates whether the forecast
tightens as the semester progresses, which is the expected behavior of any
reasonable forecasting model.

--- END PASTE ---

How to read this:
  - Each block is one entity (one box in the ERD).
  - PK = Primary Key, FK = Foreign Key.
  - "→ table.column" indicates a foreign-key reference.

────────────────────────────────────────────────────────────────────────────────
ENTITY: profiles
────────────────────────────────────────────────────────────────────────────────
  PK   id                UUID         (also FK → auth.users.id, cascade delete)
       email             VARCHAR      UNIQUE NOT NULL
       name              VARCHAR      NOT NULL
       course            VARCHAR      NULL — degree program code (e.g. 'BSCS')
       education_level   VARCHAR      NULL — '1st year', '2nd year', ..., 'PhD'
       role              VARCHAR      NOT NULL DEFAULT 'student'
                                       CHECK (role IN ('student','admin'))
       avatar_url        TEXT         NULL
       created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
       updated_at        TIMESTAMPTZ  NOT NULL DEFAULT now()

  Notes:
    - Auto-populated by SQL trigger on_auth_user_created when a new auth.users
      row is inserted (reads name / course / education_level / role from
      raw_user_meta_data).
    - Trigger touch_updated_at maintains updated_at on every UPDATE.

────────────────────────────────────────────────────────────────────────────────
ENTITY: programs
────────────────────────────────────────────────────────────────────────────────
  PK   id                UUID         DEFAULT gen_random_uuid()
       code              VARCHAR      UNIQUE NOT NULL — e.g. 'BSCS', 'BSIT'
       name              VARCHAR      NOT NULL — e.g. 'Bachelor of Science in
                                                       Computer Science'
       is_active         BOOLEAN      NOT NULL DEFAULT true
       created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()

  Notes:
    - Admin-managed lookup table.
    - Powers the registration dropdown for "Course".

ENTITY: courses                              (per-user subjects: CS101, MATH202)
────────────────────────────────────────────────────────────────────────────────
  PK   id                UUID         DEFAULT gen_random_uuid()

       code              VARCHAR      NOT NULL
       name              VARCHAR      NOT NULL

       color             VARCHAR      NULL — hex like '#4ECDC4'
       credits           NUMERIC(4,2) NULL
       current_grade     NUMERIC(5,2) NULL
       target_grade      NUMERIC(5,2) NULL
       created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
       updated_at        TIMESTAMPTZ  NOT NULL DEFAULT now()

────────────────────────────────────────────────────────────────────────────────
ENTITY: tasks
────────────────────────────────────────────────────────────────────────────────
  PK   id                UUID         DEFAULT gen_random_uuid()
  FK   user_id           UUID         NOT NULL → profiles.id (cascade delete)
       title             VARCHAR      NOT NULL
       description       TEXT         NULL
       completed         BOOLEAN      NOT NULL DEFAULT false
       status            VARCHAR      NOT NULL DEFAULT 'to-do'
                                       CHECK (status IN
                                       ('to-do','in-progress','completed'))
       priority          VARCHAR      NOT NULL DEFAULT 'medium'
                                       CHECK (priority IN ('low','medium','high'))
       category          VARCHAR      NULL
       due_date          TIMESTAMPTZ  NULL
       is_academic       BOOLEAN      NOT NULL DEFAULT true
  FK   course_id         UUID         NULL → courses.id (set NULL on delete)
       task_type         VARCHAR      NOT NULL DEFAULT 'other'
                                       CHECK (task_type IN
                                       ('assignment','exam','project',
                                        'reading','study','other'))
       estimated_time    INTEGER      NULL — minutes
       actual_time       INTEGER      NULL — minutes
       difficulty        VARCHAR      NULL
                                       CHECK (difficulty IN
                                       ('easy','medium','hard'))
       grade             NUMERIC(5,2) NULL — score after completion
       weight            NUMERIC(5,2) NULL — % of final grade
       smart_priority    NUMERIC(6,2) NULL — computed by client (0-100)
       urgency_score     NUMERIC(6,2) NULL — computed (0-50)
       importance_score  NUMERIC(6,2) NULL — computed (0-50)
       created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()

RELATIONSHIPS — draw these as Crow's-Foot connectors
────────────────────────────────────────────────────────────────────────────────

  programs (1) ────────── (0..N) profiles
       — One program is taken by zero-or-many students.
       — A student belongs to exactly one program (or none).
       — (Soft FK: profiles.course references programs.code by string;
          not enforced at DB level so admins can change program codes.)

  profiles (1) ────────── (0..N) courses
       — One student owns many subjects (CS101, MATH202, etc.).

  profiles (1) ────────── (0..N) tasks
       — One student creates many tasks.

  profiles (1) ────────── (0..N) calendar_events
       — One student has many calendar events.

  profiles (1) ────────── (0..N) study_sessions
       — One student logs many study sessions.

  courses (1) ────────── (0..N) tasks
       — One subject can have many tasks linked to it.
       — A task may have zero or one linked course.
       — (Personal tasks have course_id NULL.)

  tasks (1) ────────── (0..N) resources
       — One task can have many attached resources.
       — Resources cascade-delete when their parent task is deleted.

  tasks (1) ────────── (0..N) study_sessions
       — A study session may be linked to one task or none.
       — Deleting a task sets the session's task_id to NULL (kept for stats).


  HOW TO LAY OUT THE DIAGRAM:

  Place programs and profiles at the top.
  Below profiles place its four child tables horizontally:
    courses | tasks | calendar_events | study_sessions
  Then below tasks:
    resources    (cascade child of tasks)
  Connect study_sessions back up to tasks with a dashed line (optional FK).

  Use the legend:
    ─── solid line: required relationship (NOT NULL FK)
    ┄┄┄ dashed line: optional relationship (NULL-able FK)
